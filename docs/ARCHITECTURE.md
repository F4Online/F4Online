## Overview & Goals

F4Online is a multiplayer mod for Fallout 4, delivered as an F4SE C++ plugin (SDK: ianpatt f4se 0.7.8 + common, vendored under `C:/Users/Admin/F4Online/libs/`). It targets the Next-Gen runtime (1.10.980+ / 1.11.x), x64, Windows. A minimal plugin already builds: `C:/Users/Admin/F4Online/src/main.cpp` exports `F4SEPlugin_Version` + `F4SEPlugin_Load` and registers an F4SE messaging listener.

Product vision (the destination):

- An **Online** button appears on the Fallout 4 main menu, styled like the base UI.
- Clicking it enters Online mode and shows a **username** dialog.
- After username creation, a **server-browser** panel lists available servers (or none).
- Players can **locally host** their own servers, search/browse servers, and join.
- Servers declare **categories** (RP, Sandbox, …).
- Co-op in the open Commonwealth, staying as close to base game as possible — no gameplay overhauls.

This document commits to a **judge-recommended base** (the "MVP Listen-Server": shortest path to two friends moving together this week) while **grafting the long-term architecture** from the dedicated/authoritative proposals so the early code is forward-compatible and the eventual migration is additive, not a rewrite. Concretely: ship a host-authoritative listen server now that syncs *only* avatar transforms, but structure authority, message schemas, and the transport seam so that a single engine-agnostic **server-core** with a master-server-backed categorized browser drops in later without relocating where authority lives.

The first deliverable slice is exactly the user's request: the Online button + username dialog + a (stubbed) server browser; a small multiplayer hook (two localhost clients see each other's position move); and an immediate start at Vault 111 (skip the pre-war intro, Sanctuary mirror, and MQ intro quest).

## Target & Constraints

- **Runtime:** Fallout 4 Next-Gen, 1.10.980+ / 1.11.137+, x64, Windows. We do **not** build the f4se runtime DLL; F4SE installs its own Scaleform/threading/save hooks. We ship `F4Online.dll`, optional `.swf` assets, and an optional `F4Online.esp`.
- **Plugin contract (NG):** export `F4SEPluginVersionData F4SEPlugin_Version` and `bool F4SEPlugin_Load(const F4SEInterface*)`. `QueryInterface` provides Messaging, Scaleform, Papyrus, Serialization, Task, Object, Trampoline (`PluginAPI.h`).
- **Address independence is mandatory the moment a real hook lands.** The current `F4SEPlugin_Version` declares Address Library independence + NoStructs. Every raw offset in the headers (`g_player` `0x032D22E0` in `GameReferences.cpp:6`, `MoveRefrToPosition` `0x01180CD0`, `RegisterMenu`/`SendUIMessage`, `pos@0xD0`/`rot@0xC0` in `GameReferences.h:211-213`) is one-build-specific. `ScaleformMovie.h` is even marked `UNTESTED!!!` (line 6). Real hooks must resolve through the Address Library (versionlibdb); the `NoStructs` declaration must be revised the instant a game struct is touched, or it becomes a lie that crashes mismatched runtimes (per the comment in `src/main.cpp`).
- **The engine is single-threaded; game objects are not thread-safe.** `TESObjectREFR`, `Actor`, forms, the Papyrus VM, and Scaleform movie state must only be touched on the game thread; Creation Engine natives rely on `thread_local` state and crash off-thread. All off-thread work marshals back via `F4SETaskInterface::AddTask`/`AddUITask` (`PluginAPI.h:199`; dispatch in `Hooks_Threads.cpp:31-41`).
- **All UI is Flash.** Every FO4 menu, including the main menu (`Data/Interface/MainMenu.swf` — *not* `StartMenu.swf`, which is the FO3/Skyrim-era name), is a GFx/Flash `.swf` asset under `Data/Interface/`. F4SE manipulates an existing GFx DOM; it **cannot** draw base-styled, navigable widgets from C++. Authoring visible UI requires Adobe Animate/Flash CS6 + CLIK. This is unavoidable and is the true critical path.
- **Intro skip is quest/ESP-driven.** F4SE exposes no "skip quest stage" hook; the robust path is a small ESP (Start Me Up pattern).
- **Build:** CMake (`C:/Users/Admin/F4Online/CMakeLists.txt`, `BUILDING.md`), static CRT (`/MT`), vendored `f4se::f4se_common` + `common`. Any third-party lib (ENet now, GameNetworkingSockets + protobuf/OpenSSL/libsodium later) must also be built `/MT`.

## High-Level Architecture (layers: F4SE C++ DLL, Scaleform/.swf UI, optional ESP/Papyrus, networking, master/browser)

The system is layered so each layer has one job and the thread boundary is explicit. Layers 0–2 are engine-agnostic and reusable by a future dedicated-server binary; Layers 3–5 are the FO4/F4SE-specific edges.

- **Layer 0 — Common schema (`f4o_common`, header-mostly static lib, no engine/F4SE deps).** POD wire structs, a single `protocolVersion` constant, the entity-id model, (de)serialization helpers, fixed-tick math, and the versioned server-metadata struct. Shared by the client DLL today and a server-core binary later. Keeping this engine-free is what makes the Phase-4 server-core a recompile, not a rewrite.
- **Layer 1 — Net module + transport (`f4o_net`, static lib, no engine deps).** A thin `INetTransport` interface wrapping the transport (ENet now; GameNetworkingSockets later) with two channels (unreliable = state stream, reliable-ordered = events). Owns the dedicated net thread, the fixed-tick accumulator, and the two SPSC lock-free ring buffers (`net→game` inbound deltas, `game→net` outbound input). Holds **zero** game pointers — only POD structs with FormID/entity-id handles resolved later on the game thread.
- **Layer 2 — Game-thread Bridge (inside `F4Online.dll`).** The only layer that reads/writes game memory, and only on the game thread. An `AddTaskPermanent` delegate (`ITaskDelegate::Run()`, `GameThreads.h:3`) runs every main-loop iteration: it reads the local `g_player` transform, throttles, and enqueues outbound; and it drains the inbound queue and applies remote transforms via `AddTask`. Scaleform writes go via `AddUITask` (never `AddTask`).
- **Layer 3 — Scaleform C++ bridge (inside `F4Online.dll`).** `scaleform = QueryInterface(kInterface_Scaleform); scaleform->Register("F4Online", RegisterCallback)` publishes `f4se.plugins.F4Online` in every GFx movie (`PluginAPI.h:142`; `Hooks_Scaleform.cpp:446-528`). Native functions (`Host`, `Join`, `SubmitUsername`, `RequestServerList`, `ConnectToServer`, `GetState`) are registered via `RegisterFunction<T>` / `GFxFunctionHandler::Invoke` (`ScaleformCallbacks.h:41-73`). Native→Flash pushes (server rows, status) use `CreateArray`/`CreateObject`/`SetMember`/`Invoke` (`ScaleformMovie.h:55-79`) marshalled through `AddUITask`.
- **Layer 4 — Scaleform `.swf` assets (Flash-authored, unavoidable).** (a) A transparent **overlay** `.swf` that draws the Online button on the main menu via the reg2k MCM `addChild`-onto-`Menu_mc` technique; (b) one **custom-menu** `.swf` that hosts the username dialog and the server-browser panel as states. Authored in Adobe Animate/Flash CS6 with CLIK, using symbols/fonts copied from a JPEXS decompile of `MainMenu.swf`.
- **Layer 5 — Optional ESP + Papyrus (`F4Online.esp` + one quest script).** (a) A Start-Me-Up-style alternate start that disables the vanilla intro/CharGen quest and drops the player at a Vault 111 marker; (b) one quest script owning `UI.RegisterCustomMenu`/`OpenMenu`/`CloseMenu` (these are **Papyrus-only**, `PapyrusUI.cpp:21-65,362`), so C++ never depends on hardcoded `g_ui->Register` offsets. No gameplay logic lives here.
- **Master server (out-of-game service, Phase 3).** A small HTTP/JSON service that listen/dedicated servers heartbeat into and that the browser queries. Carries **no** gameplay traffic — discovery only. LAN UDP broadcast and direct connect-by-IP/code keep working if it is down.

## Network Topology & Transport (state the recommendation + why; mention the rejected alternatives briefly)

**Recommended topology (now → later):**

- **Now (MVP):** a **host-authoritative listen server** embedded in `F4Online.dll`, gated behind a host/client flag. One player picks **Online → Host**; their process opens a UDP listen socket and is both server and player 0. Friends pick **Online → Join** (direct IP or a 6-char invite code encoding IP:port). Star topology, 2–4 players, no separate binary. This is the F4MP / early-Skyrim-Together shape — and its host-authoritative desync weaknesses (host-only quests, enemies-target-host, alive-at-0-HP, item-drop drift documented for Skyrim Together) **cost us nothing**, because the MVP syncs only avatar transforms, none of the contested state.
- **Later (the destination, grafted from the dedicated proposals):** a single engine-agnostic **server-core** that runs identically as a standalone dedicated process *or* co-located in-process behind a loopback client. That makes "local hosting" and "join a browsed remote server" the *same* code path with **one** authority path and zero host-sim drift. To make that migration additive rather than a relocation of authority, the MVP must put the host's authority logic in a distinct module (Layer 0/1) from day one, not inline in the client.

**Transport recommendation:**

- **Now: ENet** (reliable+unreliable UDP, sequencing, fragmentation, channels, heartbeat) behind `INetTransport`. Rationale: ~3 small C files, zero external deps, trivially `/MT` into the existing CMake, and exactly the two channels we need (unreliable 20 Hz position stream + reliable-ordered join/leave/hello/username). For "two friends this week," direct-IP ENet + one port-forward (or ZeroTier/Hamachi/Radmin for NAT-blocked players) is sufficient. Fixed default UDP port (e.g. 27787), configurable.
- **Later: GameNetworkingSockets (GNS), standalone (no Steam).** This is what Tilted Evolution/Skyrim Together settled on (via TiltedConnect) for this engine class; it gives reliable+unreliable UDP, fragmentation/reassembly, AES-GCM encryption, congestion control, and ICE/STUN/TURN NAT traversal in one library — covering LAN (`ConnectByIPAddress`), direct-IP, and internet P2P. `INetTransport` lets GNS replace ENet with **no gameplay change**. Steam Datagram Relay is a Steam-build-only optional upgrade designed for behind the same abstraction; a self-hosted TURN relay is budgeted for symmetric-NAT failures (ICE is not guaranteed).

> **Open decision (see end):** if the `/MT` build of GNS + protobuf + OpenSSL/libsodium turns out to be a sub-day task, standing GNS up from Phase 0 avoids the ENet→GNS rework entirely and eliminates ENet's manual-port-forward UX (the exact failure that killed F4MP). The default plan is ENet-first for speed-to-playable, with GNS as the explicitly-planned swap.

**Rejected alternatives (briefly):** raw Asio/UDP — months re-implementing reliability/fragmentation/crypto/congestion, the exact wheel GNS already turns; **yojimbo** — lower-level, less maintained, still need discovery on top; **GNS *for the MVP only*** — its protobuf/OpenSSL/libsodium `/MT` dependency chain is precisely the infrastructure cost the speed-first base is meant to defer. **Raw P2P with manual IP + port-forward as the *product*** (F4MP, port 7779) is a UX/reliability dead end and is why that project died — we treat direct-IP as a fallback, not the shipping discovery mechanism.

## State Synchronization Model

**MVP scope: sync exactly one thing — each connected player's avatar transform.** This is the deliberate scope cut that makes host-authority free.

- **Outbound (per client):** the game-thread permanent task reads `(*g_player)->pos` (`0xD0`) + `rot` (`0xC0`) + `parentCell` (`0xB8`) / worldspace, throttles to ~20 Hz, and pushes a POD `{ clientId, cellFormId, worldspaceFormId, NiPoint3 pos, float rotZ, uint8 animFlags }` onto the `game→net` queue.
- **Relay (host):** host fans out every client's latest transform to all others — host-authoritative, last-write-wins, no validation (none needed for own-avatar pose).
- **Inbound (per receiver):** the Bridge applies remote transforms on the game thread to a per-peer avatar proxy. The first proof uses the **cheapest visible proxy** — a placed marker/static `MoveRefrToPosition`'d each update (pattern from `PapyrusObjectReference.cpp:160-165`, passing `&(*g_invalidRefHandle)` as the null target handle, `ref->parentCell`, and `CALL_MEMBER_FN(ref, GetWorldspace)()`). Upgrade to a real spawned actor per peer once the pipeline is proven. Remote transforms are interpolated between updates; the local avatar is never corrected (the local player is authoritative over their own body — no prediction/reconciliation required in the MVP).
- **Channels:** unreliable for the position stream (drop-tolerant); reliable-ordered for hello/username/join/leave.
- **Cross-cell:** if a peer's `parentCell`/worldspace differs from yours, hide their proxy.
- **Explicitly NOT synced in MVP:** animations beyond a coarse moving/idle flag, combat, health, inventory, NPCs, quests, loot, cell streaming, weather/time. We avoid Tilted's "sync by accident" simply by syncing almost nothing.

**Grafted forward-compatibility (cheap now, avoids a painful retrofit):**

- **Adopt the LocalActor/RemoteActor ownership model on day one** (Tilted precedent): every networked entity has one owner client that *authors* its state; remotes interpolate/apply. Even with only avatar transforms replicated, tagging ownership now is near-free and makes contested-state work tractable later.
- **Design the long-term sync model explicitly** (Phase 4+): server-authoritative snapshot+delta replication of an entity table on a fixed server tick (~20 Hz), unreliable channels for continuous state (position, animation graph, aim) and reliable-ordered for discrete events (inventory transfer, quest stage, NPC death, cell load), with client-side interpolation for remotes and prediction+reconciliation for the local avatar — server validating contested mutations (NPC health/death, loot/container/inventory transfers à la SkyMP, quest stages, world flags, doors/locks, dropped items). This is the model that retires the host-authoritative desync class; it is documented now so the MVP's POD structs and entity-id model are shaped toward it.
- **Cell/worldspace streaming and quest-graph determinism are explicitly deferred** (universally the least-solved area across all prior art) until the authority + discovery foundations are solid.

## Sessions, Local Hosting, Server Browser & Categories

**Local hosting (MVP).** The Online menu offers **Host** and **Join**. `f4se.plugins.F4Online.Host()` flips the DLL to host mode, opens the ENet listen socket, and makes the host player 0, immediately in-session. **Join** takes a direct `IP:port` (or invite code that encodes it). No master server, no NAT traversal in the MVP — friends use a forwarded port or a VPN overlay.

**Browser (MVP — intentionally near-stubbed but wired through the real spine).** The browser panel shows: (a) **LAN servers** discovered via a simple UDP broadcast beacon on the fixed port (zero-config, same-subnet, ~30 lines — the one "real" discovery we include); and (b) a manual **Direct Connect** row. The panel may render empty initially; that is acceptable for the first slice.

**The versioned server-metadata POD (defined now, reused unchanged forever).** This is grafted directly from the dedicated proposals so that wiring a master server later is *purely additive*:

```
struct ServerInfo {
    uint32_t protocolVersion;   // gate compatibility FIRST
    uint32_t buildVersion;
    char     name[64];
    uint8_t  category;          // enum: RP, Sandbox, Survival, PvP, Modded, Coop, Hardcore, Other
    uint16_t curPlayers;
    uint16_t maxPlayers;
    bool     passwordProtected;
    uint64_t modListHash;       // xxhash of the ordered plugin set
    uint16_t regionHint;        // optional region/ping hint
};
```

The same struct is the LAN beacon payload now, the connect handshake now, and the master heartbeat / A2S-style query reply later — **no schema migration**.

**Categories (first-class field now, real semantics later).** A `category` enum lives in `ServerInfo` from day one; the host picks one when starting; the browser displays it and offers filter chips. With no master server in the MVP, filtering is functionally cosmetic (only LAN servers exist to browse) — but the field is forward-compatible. In Phase 4 a **server-side scripting VM (Lua, the lightest; tes3mp/NV:MP/VaultMP/SkyMP all prove server-side scripting is what keeps these projects alive)** defines what RP/Sandbox/etc. actually *mean* as game-mode rules, turning categories from labels into community-authored modes. The Lua VM runs on a worker thread and schedules any engine effects through the same main-thread job queue.

**Discovery hierarchy (Phase 3 target the MVP browser is designed to plug into):** master service primary (heartbeat every 10–30 s; categorized public browser); LAN UDP broadcast secondary (zero-config, does **not** cross routers/VLANs/WiFi-isolation); direct connect-by-IP/code always-works fallback. Password-protected servers heartbeat but are **hidden** from the public list (join by code/IP), mirroring Skyrim Together. The browser greys-out / warns on `protocolVersion` or `modListHash` mismatch **before** connect so users see *why* they cannot join.

## UI Integration (Online button, username dialog, server browser — exactly how, including .swf realities)

**Reality first:** F4SE manipulates an existing GFx DOM; it cannot synthesize base-styled, gamepad/keyboard-navigable widgets from C++. Everything visible must be Flash-authored (Adobe Animate/Flash CS6 + CLIK), with symbols/fonts copied from a JPEXS decompile of `Data/Interface/MainMenu.swf`. **Flash authoring (button styling + focus/gamepad navigation) is the true cost center, not the C++ bridge.**

**(1) Online button — overlay technique, NOT a `MainMenu.swf` replacement.** Replacing `MainMenu.swf` wholesale conflicts with other UI mods and breaks on updates. Instead, use the reg2k MCM/f4mcm overlay pattern:

- In `F4SEPlugin_Load`, register the native bridge: `scaleform->Register("F4Online", RegisterCallback)`. F4SE then builds `f4se.plugins.F4Online` in every movie and fires the `F4SE::Initialized` AS event / `onF4SEObjCreated` (`Hooks_Scaleform.cpp:496-527`).
- The overlay `.swf`'s ActionScript, on `F4SE::Initialized`, detects the main menu via `root.loaderInfo.url == "Interface/MainMenu.swf"`, creates a `flash.display.Loader`, loads the overlay, and `root.Menu_mc.addChild`s an Online button styled from copied CLIK symbols.
- The button's click calls `f4se.plugins.F4Online.OpenOnline()` (a native `GFxFunctionHandler` we register), which lands in C++.

**(2) Username dialog + (3) Server browser — one custom menu, two states.** Author one custom-menu `.swf` (panel + `TextField` + OK/Cancel for the username; a scrollable CLIK list for the browser). Register it via `UI.RegisterCustomMenu(name, menuPath, rootPath, MenuData)` — a **Papyrus-only** function (`PapyrusUI.cpp:21-37,362`) handled by the thin ESP's quest script — with flags `kFlag_PausesGame | kFlag_UsesCursor` and extended flag `kExtFlag_InheritColors` so it inherits base-UI theming for free (`CustomMenu.cpp:42-46`). Open/close via `f4se.OpenMenu(...)` (built into CustomMenu, `CustomMenu.cpp:76-87`) or `UI.OpenMenu`.

- The username field calls `f4se.AllowTextInput(true)` on focus and `false` on blur (`Hooks_Scaleform.cpp:116-126`) so the field actually receives typed characters.
- **Flash → native:** all calls go through `f4se.plugins.F4Online.*` (`SubmitUsername`, `RequestServerList`, `ConnectToServer`, `HostServer`, `GetState`).
- **Native → Flash:** server rows built with `CreateArray` + per-row `CreateObject`/`SetMember`/`PushBack` then `Invoke` (the `F4SEScaleform_GetDirectoryListing` pattern), marshalled **only via `AddUITask`**.
- **Translation gotcha:** `.swf` text starting with `$` is a `BSScaleformTranslator` key (`ScaleformTranslator.h`); avoid a leading `$` or ship a translation file.

**De-risk strategy (hard gate, grafted from all three proposals):** ship a **placeholder unstyled `.swf` first** (plain button, raw `TextField`, text list) and make the first UI milestone a literal acceptance test — *one click in the overlay reaches a registered `f4se.plugins.F4Online` native function (a log line)* — **before** any styling or menu work. Then swap in CLIK art with no C++ change. GFx lifetime hygiene: create `GFxValue`s from the callback's `movieRoot`, never cache a `movieRoot` across menus, and decrement one-shot listener handler refcounts after `addEventListener` (as `PapyrusUI` `UI_LatentLoad` does).

## Skip-Intro / Start at Vault 111

**Primary, robust path — ship a minimal `F4Online.esp` (Start Me Up / alternate-start pattern).** Disable the vanilla start quest (MQ102/MQ101) and the CharGen intro, and place the player directly at a Vault 111 start marker, skipping the pre-war sequence, the Sanctuary mirror, and "Out of Time." This is data-driven, version-independent, and exactly what every shippable alt-start mod does. There is **no clean pure-C++ skip** — the intro is quest/ESP-driven and F4SE exposes no quest-stage fast-forward hook. The MVP accepts a bare default-appearance start in Vault 111 to keep scope minimal; full CharGen can return later.

**Documented C++ fallback (only if avoiding an ESP — fragile, not the path).** Subscribe to `kMessage_NewGame` (dispatched by `TESQuest::NewGame_Hook`, `Hooks_SaveLoad.cpp:69-77`, which passes the CharGen `TESQuest*`). Then, on the first valid `g_player` tick after worldspace load, resolve the Vault 111 marker via `LookupFormByID` (`GameForms.h:36-37`) and call `MoveRefrToPosition(player, &(*g_invalidRefHandle), targetCell, GetWorldspace(), &pos, &rot)` exactly as `PapyrusObjectReference.cpp:160-165`. This is brittle (you must also suppress the vanilla intro yourself, find a valid marker/cell, and time it post-load), so it is a fallback only. There is no console `coc` exposed — only `g_consoleHandle`/`GetCurrentConsoleRef` (`PapyrusGame.cpp:18-28`); `coc` must be emulated via the move API.

## Threading & Game-Loop Integration

The single existential correctness constraint: **the net thread holds zero game pointers, and every engine/Papyrus/Scaleform touch happens on the game thread.** Enforce it structurally, not by convention.

- **Net thread (one `std::thread`):** owns the transport (ENet host/peer; later the GNS poll group) and the fixed ~20 Hz tick accumulator. Does all socket I/O, packet decode/encode, interpolation bookkeeping, and (later) crypto/decompression. Never dereferences a game pointer. Do **not** couple the sim tick to render framerate.
- **Two SPSC lock-free ring buffers** carry POD structs across the boundary (`net→game` inbound deltas, `game→net` outbound input). Two queues are mandatory because tasks are fire-and-forget with **no** synchronous return value (`Hooks_Threads.cpp:35-41`).
- **Game thread:** `AddTaskPermanent` (`PluginAPI.h:199-210`) drives a per-frame `ITaskDelegate` that (a) reads the local `g_player` transform, throttles, enqueues outbound; (b) drains the inbound queue and applies remote transforms. The permanent task runs every main-loop iteration inside `MessageQueueProcessTask_Hook` (`Hooks_Threads.cpp:31-33`) — i.e. per-frame, so **throttle inside `Run()`**, do not assume a fixed cadence.
- **Routing discipline:** gameplay writes use `AddTask`; Scaleform/list writes use `AddUITask` (`Hooks_Threads.cpp:61-74`). Mixing them races the UI thread.
- **`Run()` must be cheap:** apply only pre-decoded deltas. No parsing, no crypto, no blocking — a slow `Run()` stalls rendering.
- **Gating:** null-check `*g_player` (it is null at main menu / during load) and gate on game-loaded (`kMessage_GameLoaded` / `kMessage_PostLoadGame`) + `parentCell != null` before any read/move, to avoid main-menu/loading crashes.
- **Build the spine before any hook.** The presence-beacon proof (Step A below) exercises net-thread → ring buffer → `ITaskDelegate` → `AddUITask` end-to-end with **zero** game-state hooks and zero Address Library risk, so a sync bug can never corrupt the bridge. Only after it works do real game hooks land — each behind the Address Library from its first commit.

## Persistence (co-save)

Two stores, deliberately separated.

- **F4SE co-save (`F4SESerializationInterface`, `PluginAPI.h:153-180`) — per-save LOCAL state only.** In `F4SEPlugin_Load`: `QueryInterface(kInterface_Serialization)`, `SetUniqueID('F4ON')`, and `Set{Save,Load,Revert}Callback`. Store the chosen **username/identity token** and the **last-joined server (IP/invite code)**: `OpenRecord(type, version)` + `WriteRecordData` on save; loop `GetNextRecordInfo` + `ReadRecordData` on load (mirroring `InternalSerialization.cpp` `Core_Save/LoadCallback`). **Any persisted FormID/handle MUST be remapped on load via `ResolveFormId`/`ResolveHandle`** (`Serialization.cpp:277-333`) — load order shifts the modIndex high byte, including the `0xFE` light-plugin range — or references silently corrupt.
- **Critical caveat (from recon):** the co-save fires **only on the vanilla save/load path** (`Hooks_SaveLoad.cpp`). A pure multiplayer session that never saves won't trigger it. Therefore the MVP **also mirrors** username + last server to a plain JSON/INI under `Documents/My Games/Fallout4/F4SE/F4Online/` as the source of truth for menu prefill; the co-save is a nice-to-have, not the live store.
- **Authoritative live multiplayer state** (who's connected, transforms) is in-memory in the MVP and, in Phase 4, lives in the **server-core's own out-of-band store** (flat-file/embedded-DB snapshot for persistent dedicated servers, à la tes3mp state save/load) — never the co-save.
- **Versioning:** the single `protocolVersion` constant gates the connect handshake, the (future) master heartbeat schema, **and** the co-save record schema. Bump it deliberately; reject mismatches loudly.

## Security & Anti-Cheat (lightweight)

Honest scope: friends-trust co-op now, server-authoritative validation as the project scales.

- **Handshake gating from the first networked build:** check `protocolVersion` **first**, then `buildVersion`, then `modListHash` (xxhash of the ordered plugin set). Reject loudly; the browser greys-out/warns before connect. The recon is emphatic that silent mod divergence ("sync by accident") becomes intractable to debug if this gate lands late — it is cheap now and prevents that, and makes categories filterable for real over LAN immediately.
- **Crash-hardening (MVP, because the host is a real player's game):** length-check and bounds/rate-limit every decoded packet on the net thread so a malformed or flooding peer can't crash the host. Sanitize and length-cap usernames before they reach Flash or the wire.
- **Passwords:** optional shared session password compared host-side; password-protected servers are hidden from any public list (join by code/IP).
- **MVP non-goals:** no encryption, no anti-cheat, no server-side movement validation — host-authoritative + friends-only makes those unnecessary for what is synced.
- **Later (with GNS / server-core):** AES-GCM transport encryption and authenticated connections for free from GNS; **server-authoritative validation becomes the core anti-cheat lever** — the server validates all contested mutations (inventory/container/loot transfers and reloot resets à la SkyMP, NPC death, quest stages) and rate-limits per connection; clients are authoritative only over their own avatar pose. Steam Datagram Relay (Steam-build-only) hides host IPs and adds DoS protection; signed identity tokens enable persistent accounts. Heartbeat fields from the master are treated as untrusted display metadata, never as authority.

## First Implementation Slice (precise, step-by-step)

The slice has five deliverables. The two networking sub-steps come first because Step A carries zero Address Library risk and builds the spine everything else rides on. The UI Flash work proceeds in parallel against placeholder assets.

### Slice 0 — UI bridge proof (do this before any styling)

1. In `F4SEPlugin_Load`, `QueryInterface(kInterface_Scaleform)` and call `scaleform->Register("F4Online", RegisterCallback)`.
2. In the callback, `RegisterFunction<T>` a single native `OpenOnline()` that writes a line to `F4Online.log`.
3. Author a minimal overlay `.swf`: on `F4SE::Initialized`, check `root.loaderInfo.url == "Interface/MainMenu.swf"`, `Loader`-load the overlay, `root.Menu_mc.addChild` a plain (unstyled) button calling `f4se.plugins.F4Online.OpenOnline()`.
- **Acceptance:** clicking the placeholder button on the main menu produces the log line. *Proves Flash → native, the real cost center, before any art.*

### Step A — Presence beacon (network spine, zero game hooks)

1. Add ENet to CMake (`/MT`); implement `INetTransport` + `EnetTransport`.
2. Start the net thread; implement Host/Join over localhost; the only replicated state is connected-player count.
3. Flow it through the real spine: net thread accepts a connection → pushes a `PlayerCount` delta onto the `net→game` ring buffer → Bridge marshals it via **`AddUITask`** into the overlay text (or `F4Online.log` for v0).
- **Acceptance:** two FO4 instances on localhost; the second connecting increments the count surfaced on the first. *Proves net-thread → lock-free queue → `ITaskDelegate` → UI end-to-end; loads on any NG runtime.*

### Step B — Small multiplayer hook (first real game delta, behind the proven spine)

1. Switch `F4SEPlugin_Version` off `NoStructs`; resolve **every** address via the Address Library from this commit on.
2. `QueryInterface(kInterface_Task)`; `AddTaskPermanent` a delegate that reads `(*g_player)->pos` (`0xD0`) / `rot` (`0xC0`) — null-check `g_player`, gate on `parentCell != null` — throttles to 20 Hz, and enqueues outbound.
3. Host fans out transforms; each receiver, via `AddTask`, applies the remote transform to one per-peer proxy with `MoveRefrToPosition(proxy, &(*g_invalidRefHandle), parentCell, GetWorldspace(), &pos, &rot)`. Start the proxy as a placed marker/static; interpolate between updates.
- **Acceptance:** two FO4 instances on one LAN/localhost, each player sees the other's avatar move around Vault 111 / the Commonwealth in near-real-time. *This is the user's "small multiplayer hook."*

### Step C — Username dialog

1. Add `SubmitUsername(name)` to the `f4se.plugins.F4Online` bridge.
2. Author the custom-menu `.swf` username state (panel + `TextField` + OK/Cancel); register it via `UI.RegisterCustomMenu` from `F4OnlineQuest.psc` with `kFlag_PausesGame | kFlag_UsesCursor | kExtFlag_InheritColors`. Field calls `f4se.AllowTextInput(true)` on focus / `false` on blur.
3. On OK, AS calls `SubmitUsername`; C++ sanitizes/length-caps and persists to JSON + co-save (`SetUniqueID('F4ON')`).
- **Acceptance:** Online → username dialog appears → typed name is captured in C++ and survives a relaunch (prefilled from JSON).

### Step D — Server-browser stub

1. Add `RequestServerList(filter)` and `ConnectToServer(...)` to the bridge.
2. Author the browser state of the custom menu (CLIK list + category filter chips + Host/Direct-Connect rows). It may render empty initially.
3. Wire LAN UDP broadcast discovery to populate rows via `CreateArray`/`CreateObject`/`SetMember`/`Invoke` through `AddUITask`; rows carry the `ServerInfo` POD (incl. `category`, `modListHash`).
- **Acceptance:** opening the browser shows the panel; a LAN host appears as a row with its category and is joinable; mismatched `protocolVersion`/`modListHash` rows are greyed out with a reason.

### Step E — Skip intro / start at Vault 111

1. Build `F4Online.esp` in the Creation Kit (Start-Me-Up pattern): disable the vanilla start/CharGen quest; place a Vault 111 start marker.
- **Acceptance:** a new game lands the player directly in Vault 111 with no pre-war intro, Sanctuary mirror, or MQ102 intro quest.

## Risks & Open Questions

- **Flash/.swf authoring is the true critical path, not the netcode.** A base-styled, gamepad/keyboard-navigable Online button + username field + list requires Adobe Animate/Flash CS6 + the CLIK/F4CF interface kit and real ActionScript, with symbols from a JPEXS decompile of `MainMenu.swf`. *Mitigation:* placeholder unstyled `.swf` first (Slice 0 acceptance test), restyle later with no C++ change. If Flash skills are absent, this single item can dominate the schedule.
- **Host-authoritative is a real architectural dead-end for the full vision.** Retrofitting server-authority is the expensive path the dedicated proposals avoid; the MVP defers it to Phase 4. *Mitigation:* the `INetTransport` seam, the engine-free `f4o_common`/`f4o_net` layers, the LocalActor/RemoteActor model, and the `ServerInfo` struct are all in place from day one so the migration is "compile the core into a co-located process + add a loopback client," not a relocation of authority. The blast radius is bounded, but the transition is genuine work — not free.
- **NG version fragility the instant a hook lands.** `g_player`, `MoveRefrToPosition`, pos/rot offsets, `RegisterMenu`/`SendUIMessage` are one-build offsets; `ScaleformMovie.h` is flagged UNTESTED. *Mitigation:* keep Step A address-free; gate Step B behind the Address Library and revise `F4SEPlugin_Version` (off `NoStructs`) from its first commit.
- **Threading violations are instantly fatal.** Any game-object access from the net thread, any heavy work in `ITaskDelegate::Run()`, or `AddTask`-vs-`AddUITask` misrouting crashes or stalls. *Mitigation:* net thread holds zero game pointers; POD-only across the two queues; build and validate the spine (Step A) before any hook exists.
- **Proxy-actor-per-peer is finicky** (3D not loaded, cell mismatch, cleanup). *Mitigation:* start with a placed static/marker; upgrade to a real actor once the transform pipeline is proven; always null-check `parentCell` and hide cross-cell peers.
- **NAT/connectivity for non-LAN friends** in the MVP needs a forward or VPN overlay; symmetric NAT breaks ICE later too. *Mitigation:* accepted for MVP; real fix is GNS + master/TURN in Phase 3; TURN hosting is a budgeted cost.
- **Infrastructure cost/SPOF (Phase 3+):** master server (and TURN relay) are services the project must host. *Mitigation:* LAN + direct-IP must remain fully functional when the master is down.
- **Co-save only fires on vanilla save/load.** *Mitigation:* mirror local prefs to JSON; keep live state out of the co-save.
